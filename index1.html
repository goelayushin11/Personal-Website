Project Overview

AI Project Assistant

A three-agent system designed to eliminate the manual overhead of post-meeting follow-up. Each agent's output feeds directly into the next — with human approval gates keeping the PM in control throughout.

What Problem It Solves

Scaling the PM function to match a fast-growing business was becoming a bottleneck. The team was struggling to manage workload and prioritize effectively — and on-time task completion rates were falling. The manual overhead of post-meeting follow-up was consuming hours that should have gone to strategic work.

Technical Approach

Claude Cowork
Built using Claude's Projects feature with the Sonnet 4.6 model. Projects maintained full context across sessions — no re-explaining, no lost state.
Modular agent architecture
Each agent has a single responsibility. The folder nesting ensures one agent's output is the next agent's exact input — no reformatting, no handoff errors.
Human-in-the-loop design
Two mandatory approval gates prevent any agent from taking outbound actions autonomously. Automation with judgment, not instead of it.
Markdown-first data exchange
Agents communicate via structured markdown, not direct API calls to each other. This makes the system portable, inspectable, and easy to debug.

Key Learnings

👤 What only a human can own
Strategic task capture
AI transcribers are 50–60% effective from a PM's view. They capture what was said — not what was assumed. Action items that were never spoken still need to be surfaced. That's a PM's job.
Due date judgment
Dates are rarely stated in meetings. A PM assigns them from an interconnected web of context — dependencies, capacity, stakeholder expectations — that no agent can infer with any degree of accuracy.
Escalation & prioritization
Deciding what matters, what to flag, and what to let slide. That judgment stays human.
⚡ What can be agentified
Mechanical task processing
Transcript retrieval, action item extraction, Asana population — 1–2 hours of daily work, fully automated.
Personalized reminders at scale
Consistent, timely Slack DMs per assignee — without the PM manually chasing anyone.
Highly effective knowledge management
All context from meetings and channels now lives in one place. The agent becomes a single queryable database — virtually eliminating constant switching between tabs, windows, and tools.
✦

My takeaway: Agentification isn't just about efficiency. It's also about the clarity it forces you to build into your systems. When you have to explain a process precisely enough for an AI to execute it, you discover how much ambiguity you were living with.

The agents handle the repeatable admin work. I handle the judgment and the strategy. That effective division of labor is what makes the whole system work.

Project Overview

Enterprise Portfolio Database

I am not a coder. I vibe coded a simple, yet full-stack internal tool using Claude chat and hosted it on Netlify.

What Problem It Solves

As the company portfolio grew, there was no single shared reference for what each product actually offered. Internal teams constantly pinged each other for basic answers, and new team members had no path to get up to speed quickly. The cost was slow onboarding, repeated questions, and a non-scaleable knowledge management system.

🔍
Features — Searchable, editable (password protected), and includes a chatbot to serve basic comparisons across courses.
🖥
Frontend — HTML, CSS, and JavaScript interface with search, filtering by course, and tab-based navigation across two portfolio types.
🔗
Backend + Live Sync — Google Apps Script connected to Google Sheets as the live data source. Any edit writes back to the sheet automatically — bi-directional sync.

How It Was Built

Claude
Claude wrote every line of code — frontend, backend, and sync logic. I drove every decision: what to build, how to structure it, what the UX should feel like.
Google Sheets as the data layer
The sheet is the single source of truth. The Apps Script backend pulls from it on load and pushes changes back on edit — keeping the wiki always current without a separate database.
Shipped to Netlify
The final product is live and used company-wide in active onboarding flows.

Key Learnings

👤 Only a human could do
Decide what's worth building
Identifying the right problem, scoping it correctly, and deciding what to cut — that required knowing the team's actual workflow from the inside.
Define structure and scope
Which sections, which rows, what belongs in the wiki vs. not. Every structural call required product judgment, not just information.
Drive adoption
Getting the wiki into onboarding flows required organizational credibility — not code. That's a human job.
Know when something is wrong
Recognizing that the sync was broken or the data structure was off — even without being able to fix it yourself — is a judgment skill AI doesn't have.
⚡ Better with AI
All the code
HTML, CSS, JavaScript, and Google Apps Script — written, revised, and debugged entirely by Claude across a multi-session build.
Debugging complex logic
Row ID collisions, hyperlink extraction from rich-text cells, bi-directional sync edge cases — all resolved in conversation,
Reformatting and generating data
CSV structure, Apps Script backend configuration, applying consistent formatting across 110+ rows — fast, accurate, no manual effort.
Holding context across a long build
Claude maintained the full technical context across multiple sessions — so decisions made in session 1 didn't have to be re-explained in session 5.
AI collapsed the execution gap — but couldn't replace the product thinking. Every meaningful decision — what to build, who it's for, what good looks like — required a human. Everything that was slow, repetitive, or technically outside my skill set was faster and better with AI.
